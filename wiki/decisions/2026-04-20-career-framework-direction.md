# Career Framework Direction

## Decision

Use the [[concepts/remote-guild-quest-system|Remote Guild Quest System]] as the career framework for becoming a [[concepts/remote-first-frontend-fullstack-engineer|remote-first frontend-based fullstack engineer]].

The system combines a what-if lab with a light RPG skin: weekly experiments create artifacts, artifacts become career loot, and hiring checkpoints become boss fights rather than sources of dread.

## Context

The user wants to get hired by a remote-first company as a frontend-based fullstack engineer, but wants the process to be fun rather than overly serious or anxiety-driven.

## Options considered

| Option | Pros | Cons | Outcome |
| --- | --- | --- | --- |
| Remote Guild RPG | Fun, sticky, emotionally light | Could drift into game skin without career proof | Adopt as framing layer |
| What-If Lab | Strong connection between experiments and evidence | Could feel dry without playful language | Adopt as operating core |
| Studio / Creator Mode | Produces polished public artifacts | May create public-performance pressure too early | Keep optional |
| Productivity tracker | Easy to measure | Too grim; risks guilt loops | Reject |

## Rationale

Remote-first hiring rewards more than technical skill. The user should show how they communicate, reduce ambiguity, document tradeoffs, and ship product slices. A playful quest system keeps the process sustainable while every quest still produces concrete hiring evidence.

## Consequences

- The first quest should focus on Remote Trust.
- Weekly work should end in artifacts, not only study notes.
- Rejection and feedback should be treated as telemetry for the next quest.
- Public sharing remains optional until the user chooses otherwise.
